## Summary
GEO_ERP_System is a clean, well-structured Arabic-first demo ERP; CLAUDE.md is accurate and the backend code is disciplined (parameterized queries, whitelisted generic CRUD, balanced-journal validation, immutable event log, no console.log in src, correct RTL/lang on index.html). The dominant risk is operational, not code quality: the deploy script publishes a fully open API (no auth, no CORS) on port 80 of a public IP, where the generic `/api/:resource` router grants anyone read/write/DELETE on every business table including accounting. Secondary issues are repository hygiene — built `dist/`, ~11 MB of pasted screenshots (3 byte-identical duplicates), and duplicate planning docs sit in the tree, and the project is not initialized as a git repo despite having a `.gitignore`. The flagged "destructive debug actions" are actually simulated/cosmetic and the debug module is route-locked, so that lead is downgraded. Overall grade: C

## Critical
### [C1] Public deploy exposes an unauthenticated full-CRUD API on all business data
- Where: deploy/deploy.sh:89-105 (nginx `server_name 187.124.183.65`, `listen 80`, proxy → Node) + server/index.ts:16-28 (no auth/CORS middleware) + server/routes/resource.ts:85-251 (generic GET/POST/PUT/DELETE over every REGISTRY table)
- Evidence: deploy binds nginx port 80 to a public IP and proxies to the Node app; any client can `GET/PUT/DELETE /api/employees`, `/api/journal_entries`, `/api/banks`, `/api/accounts`, etc. with no credentials. The header role switcher is cosmetic only.
- Impact: Anyone who reaches the host can read salaries, payroll, bank account numbers and balances, and full accounting ledgers, and can alter or delete any record. For a real construction-group ERP this is total data exposure plus data-loss/tampering risk. "Demo, no auth by design" is acceptable on localhost but not on a public IP.
- Fix: Do not deploy this build to a public host as-is. If a demo must be reachable, gate it (HTTP basic auth in the nginx site, an allowlist/VPN, or a real auth layer on `/api`) and bind `HOST=127.0.0.1` (already set in the systemd unit) plus restrict nginx to trusted IPs. Document the "no auth" assumption prominently in deploy/README.md.

## High
### [H1] Project is not a git repository — no version control or history
- Where: project root (no `.git` directory; `.gitignore` present at root)
- Evidence: `[ -d .git ]` returns false; CLAUDE.md line 38 confirms "Not a git repo (has .gitignore but no .git initialized)".
- Impact: No history, no branching, no rollback, no review trail for a multi-module ERP with real accounting logic. Any mistaken edit or `npm run seed` reset is unrecoverable. The carefully written `.gitignore` (which excludes node_modules, dist, and the DB/WAL) is doing nothing.
- Fix: `git init` at the root, commit the current tree (the `.gitignore` already excludes the heavy/generated artifacts), and remove the committed `dist/` and `Pasted image*.png` files before the first commit so they never enter history (see M1/L1).

## Medium
### [M1] Built output and ~11 MB of screenshots live in the source tree
- Where: project root — `dist/` (assets, index.html, qeg-logo.png) and `Pasted image.png` + `Pasted image (2..10).png`
- Evidence: `dist/` is present though it is generated by `npm run build`; the ten PNGs total roughly 11 MB. `Pasted image (6).png`, `(7).png`, `(8).png` are byte-identical (md5 `2571c7a0…`, 1067294 bytes each) — pure duplicates.
- Impact: Tree bloat and, once git is initialized, permanent history bloat. `dist/` will drift from source and can be served stale. The screenshots have no role in the running app.
- Fix: Delete `dist/` (it is already in `.gitignore`; regenerate with `npm run build`). Remove the pasted screenshots from the tree (or move them out of the project); at minimum delete the two duplicate copies. Do this before `git init`.
### [M2] Duplicate planning documentation under files/
- Where: files/erp-system-plan/erp-plan/ duplicates the files/*.md planning docs
- Evidence: `files/` holds `00_ERP_OVERVIEW.md … 09_SKILLS_GUIDE.md` plus a nested `erp-system-plan/erp-plan/` copy; CLAUDE.md line 45 also notes the duplication.
- Impact: Two copies of the same spec drift apart; readers cannot tell which is canonical.
- Fix: Keep one canonical `files/*.md` set and delete the `files/erp-system-plan/` subtree.
### [M3] Request-body limit (2mb) is smaller than the proxy limit (5m); no global API error handler
- Where: server/index.ts:17 (`express.json({ limit: '2mb' })`) vs deploy/deploy.sh:94 (`client_max_body_size 5m`); error handling is per-route only
- Evidence: nginx accepts up to 5 MB but Express rejects bodies over 2 MB, so a 2–5 MB request fails at the app layer with a default (HTML) error rather than the JSON shape the SPA expects. There is no Express error-handling middleware; each handler wraps its own try/catch and unhandled throws fall through to Express's default handler.
- Impact: Confusing/ inconsistent error responses near the size boundary; any code path that throws outside a try/catch returns an HTML stack instead of JSON `{error}`. Minor for a demo, but inconsistent.
- Fix: Align the two limits (raise Express to 5 MB or lower nginx to 2 MB) and add a small terminal error-handling middleware that always returns JSON.

## Low
### [L1] Screenshot duplicates specifically (hygiene)
- Where: project root — `Pasted image (6).png` / `(7).png` / `(8).png`
- Evidence: identical md5 and byte size (1067294 bytes) across all three.
- Impact: Wasted ~2 MB; nothing references them.
- Fix: Delete at least the two redundant copies (folded into M1).
### [L2] Hardcoded server IP baked into deploy defaults
- Where: deploy/deploy.sh:19 (`SERVER_NAME="${SERVER_NAME:-187.124.183.65}"`) and deploy/README.md
- Evidence: a specific public IP is the default nginx `server_name`.
- Impact: Couples the repo to one host; copy/paste deploys may target the wrong server. (It is overridable via env var, hence Low.)
- Fix: Default `server_name` to `_` (catch-all) and document overriding `SERVER_NAME`; drop the literal IP from README.
### [L3] better-sqlite3 calls are synchronous on the request path
- Where: server/routes/*.ts (all DB calls), server/db/connection.ts:13-15
- Evidence: better-sqlite3 is synchronous by design; every query blocks the single Node event loop, and some report queries (trial-balance, vouchers) scan all journal lines.
- Impact: Fine at demo scale, but a few concurrent heavy report requests will serialize and stall the whole API.
- Fix: Acceptable for now; if usage grows, add indexes on `journal_lines(entry_id, account_code)` and cap/paginate report ranges. No action needed for the demo.
### [L4] Debug module flagged as destructive is actually simulated and route-locked
- Where: src/modules/debug/ActionsPanel.tsx:65-93, QueriesPanel.tsx, ApiPanel.tsx:9-26; src/config/nav.ts:40-43
- Evidence: ActionsPanel buttons use `setTimeout` and a visible "These are simulated demo operations and do not affect the live server" note; QueriesPanel renders seed data; ApiPanel only probes three read-only endpoints (`/health`, `/companies`, `/dashboard`). `UNLOCKED_MODULES` is `{dashboard, accounting}`, so `/debug` renders `LockedPage`.
- Impact: No real destructive capability and not user-reachable in the demo gating. Recorded only to correct the prior lead.
- Fix: None required; if the debug module is ever unlocked, ensure the simulated actions stay simulated and that ApiPanel is not extended to mutate data.

## Optimizations
- Add the missing composite indexes noted in L3 if report endpoints get slow; the report SQL itself is well-formed and fully parameterized.
- `server/routes/resource.ts` is a genuinely good generic-CRUD pattern: identifiers are constrained to static REGISTRY keys and PRAGMA-discovered columns (`cols.has(...)`), sort/order are validated against columns, and LIMIT/OFFSET are coerced with `Math.floor(Number(...))`; user values always bind with `?`. The identifier-injection surface flagged in the leads was re-verified across LIST/GET/POST/PUT/DELETE and is sound — keep this invariant (never interpolate a raw request value as an identifier) if the file is extended.
- Consider a build step or CI check that fails if `dist/` or `*.png` screenshots reappear in the tree.

## Quick Wins
1. Delete `dist/` and the pasted screenshots (including the 3 identical duplicates) from the tree, then regenerate `dist/` with `npm run build` only when needed (~10 min, addresses M1/L1).
2. `git init` + first commit now that `.gitignore` is in place, after removing the artifacts above (~15 min, addresses H1).
3. Default `SERVER_NAME` to `_` and align the Express/nginx body limits to one value (~15 min, addresses L2 and M3).

## Suggested Next Steps

2. Initialize git and commit the cleaned tree (remove `dist/`, screenshots, and the duplicate `files/erp-system-plan/` first).
3. Add an Express JSON error-handling middleware and align the request-size limits.
